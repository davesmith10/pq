# obi-wan Algorithm Reference

This document precisely describes the two hybrid cryptographic file formats produced by
**obi-wan**: **OBIWAN** (hybrid KEM encryption) and **HYKE** (hybrid KEM + signature).
An implementer should be able to reproduce either format from this document alone.

---

## 1. Overview

Both formats combine a classical (pre-quantum) algorithm with a post-quantum algorithm so
that security holds as long as either algorithm remains unbroken.

| Format | Operation | Classical layer | PQ layer |
|--------|-----------|----------------|----------|
| OBIWAN | Encrypt / Decrypt | X25519 or ECDH (P-256/384/521) | Kyber512/768/1024 |
| HYKE   | Sign / Verify (encrypt-then-sign) | same KEM + Ed25519 or ECDSA | same KEM + Dilithium2/3/5 |

Both operations require a **tray** — a named bundle of key slots generated by **scotty**.
Trays come in four hybrid profiles (level0 and level1 are classical-only and PQ-only; they
support OBIWAN but cannot be used with HYKE, which requires all four slots).

---

## 2. Tray Types and Algorithm Mapping

| Tray level   | KEM-classical | KEM-PQ     | Sig-classical | Sig-PQ      | HYKE? |
|--------------|---------------|------------|---------------|-------------|-------|
| level0       | X25519        | —          | Ed25519       | —           | no    |
| level1       | —             | Kyber512   | —             | Dilithium2  | no    |
| level2-25519 | X25519        | Kyber512   | Ed25519       | Dilithium2  | yes   |
| level2       | P-256         | Kyber512   | ECDSA P-256   | Dilithium2  | yes   |
| level3       | P-384         | Kyber768   | ECDSA P-384   | Dilithium3  | yes   |
| level5       | P-521         | Kyber1024  | ECDSA P-521   | Dilithium5  | yes   |

level2-25519 is the default tray type.

The slot-selection rule when reading a tray is:
- **KEM-classical**: first slot whose `alg_name` ∈ {X25519, P-256, P-384, P-521}
- **KEM-PQ**: first slot whose `alg_name` starts with "Kyber"
- **Sig-classical**: first slot whose `alg_name` ∈ {Ed25519, ECDSA P-256, ECDSA P-384, ECDSA P-521}
- **Sig-PQ**: first slot whose `alg_name` ∈ {Dilithium2, Dilithium3, Dilithium5}

## 3. OBIWAN Format (Encrypt / Decrypt)

### 3.1 Wire Layout

All multi-byte integer fields are **big-endian**.

```
Offset   Size     Field
------   ----     -----
     0      8     Magic: ASCII "OBIWAN01"
     8      1     KDF byte:    0x00 = SHAKE256,     0x01 = KMAC256
     9      1     Cipher byte: 0x00 = AES-256-GCM,  0x01 = ChaCha20-Poly1305
    10      4     CT_classical_len  (uint32)
    14      N     CT_classical      (N = CT_classical_len bytes)
  14+N      4     CT_pq_len         (uint32)
  18+N      M     CT_pq             (M = CT_pq_len bytes)
18+N+M      R     Payload: nonce(12) || tag(16) || ciphertext(R-28 bytes)
```

`R` is the total payload byte count.  The full wire blob is base64-encoded and wrapped in
PEM-style armor (see §3.6).

### 3.2 Classical KEM Encapsulation

**X25519 (level2-25519)**

1. Generate an ephemeral X25519 keypair (eph_sk, eph_pk).
2. CT_classical = eph_pk (32 bytes, raw little-endian Curve25519 u-coordinate).
3. SS_classical = X25519(eph_sk, recipient_pk) — 32-byte shared secret.

**ECDH P-256 / P-384 / P-521 (level2 / level3 / level5)**

1. Generate an ephemeral EC keypair on the appropriate curve.
2. CT_classical = ephemeral public key as an uncompressed point (0x04 || X || Y):
   - P-256: 65 bytes; P-384: 97 bytes; P-521: 133 bytes.
3. SS_classical = x-coordinate of ECDH(eph_sk, recipient_pk):
   - P-256: 32 bytes; P-384: 48 bytes; P-521: 66 bytes.

### 3.3 PQ KEM Encapsulation (Kyber)

Uses the CRYSTALS-Kyber reference implementation.

1. CT_pq, SS_pq = Kyber{512|768|1024}.Enc(recipient_pk).
   - Kyber512: CT = 768 B, SS = 32 B.
   - Kyber768: CT = 1088 B, SS = 32 B.
   - Kyber1024: CT = 1568 B, SS = 32 B.

### 3.4 Key Derivation

Both modes produce a 32-byte symmetric key from (SS_classical, SS_pq, CT_classical, CT_pq).

The `len32(x)` helper encodes `len(x)` as a 4-byte big-endian unsigned integer followed
immediately by the bytes of `x`.

#### SHAKE256 mode (default)

```
buf = len32(SS_cl) || SS_cl
   || len32(SS_pq) || SS_pq
   || len32(CT_cl) || CT_cl
   || len32(CT_pq) || CT_pq

key = SHAKE256(buf, output_length=32)
```

#### KMAC256 mode

```
key_param  = SS_cl
msg        = len32(SS_pq) || SS_pq
          || len32(CT_cl) || CT_cl
          || len32(CT_pq) || CT_pq
custom_str = "hybrid-kem-file-encryption-v1"   (30 bytes)

key = KMAC256(key=key_param, msg=msg, output_length=256 bits, custom=custom_str)
```

> **XKCP note**: the XKCP `KMAC256()` function takes all length arguments in **bits**,
> not bytes.  Convert `key_param.size() * 8`, `msg.size() * 8`, and `custom_str.size() * 8`.

### 3.5 Symmetric Encryption

Both AEAD modes produce output in the layout `nonce(12) || tag(16) || ciphertext`.
The nonce is chosen uniformly at random for each encryption.
No additional data (AAD) is used.

**AES-256-GCM**
- Key: 32-byte key from KDF.
- Nonce: 12 bytes random (CSPRNG).
- Tag: 16-byte GCM authentication tag, prepended before ciphertext.

**ChaCha20-Poly1305**
- Key: 32-byte key from KDF.
- Nonce: 12 bytes random (CSPRNG).
- Tag: 16-byte Poly1305 authentication tag, prepended before ciphertext.

### 3.6 Encrypt Procedure

1. Load tray; locate KEM-classical slot and KEM-PQ slot.
2. Read plaintext from input file.
3. Perform classical KEM encapsulation → (CT_cl, SS_cl).
4. Perform Kyber encapsulation → (CT_pq, SS_pq).
5. Derive 32-byte symmetric key via the selected KDF (SHAKE256 or KMAC256).
6. Encrypt plaintext with the selected AEAD cipher → payload = nonce || tag || ct.
7. Serialise wire bytes per §3.1.
8. Base64-encode wire bytes (standard alphabet, 64-character line wrap).
9. Wrap in armor:
   ```
   -----BEGIN OBIWAN ENCRYPTED FILE-----
   <base64 lines>
   -----END OBIWAN ENCRYPTED FILE-----
   ```
10. Write armored text to output.

### 3.7 Decrypt Procedure

1. Load tray; locate KEM-classical slot (requires sk) and KEM-PQ slot (requires sk).
2. Strip armor from input; base64-decode to wire bytes.
3. Parse wire header per §3.1 to recover: KDF byte, Cipher byte, CT_cl, CT_pq, payload.
4. Perform classical KEM decapsulation → SS_cl.
   - X25519: SS_cl = X25519(rec_sk, CT_cl_as_eph_pk).
   - P-curves: SS_cl = x-coordinate of ECDH(rec_sk, CT_cl_as_eph_pk).
5. Perform Kyber decapsulation → SS_pq = Kyber.Dec(rec_sk, CT_pq).
6. Derive symmetric key with the same KDF indicated in the header.
7. Decrypt and authenticate payload (nonce || tag || ct) using the indicated cipher.
8. Write plaintext to output.

## 4. HYKE Format (Sign / Verify)

HYKE is an **encrypt-then-sign** format.  The payload is encrypted first, and both
signatures cover the encrypted payload together with the header and a context binding
over the public keys.  This prevents key-substitution attacks.

HYKE requires a tray with all four slots present (KEM-classical, KEM-PQ, Sig-classical,
Sig-PQ).  Sign needs the signing secret keys; Verify needs the KEM secret keys.

### 4.1 Wire Layout

All multi-byte integer fields are **big-endian**.

```
Offset      Size  Field
------      ----  -----
     0         4  Magic: 0x48 0x59 0x4B 0x45  ("HYKE")
     4         2  Version: 0x00 0x01
     6         1  TrayID (see §4.2)
     7         1  Flags: 0x00  (reserved)
     8         4  header_len  = 80 + N + M + P + Q  (uint32)
    12         4  payload_len = R  (uint32)
    16        16  tray_uuid   (16 raw bytes; UUID string stripped of dashes)
    32        32  salt        (32 random bytes, chosen fresh per file)
    64         4  ct_classical_len = N  (uint32)
    68         4  ct_pq_len        = M  (uint32)
    72         4  sig_classical_len = P  (uint32)
    76         4  sig_pq_len        = Q  (uint32)
    80         N  CT_classical bytes
  80+N         M  CT_pq bytes
80+N+M         P  sig_classical bytes
80+N+M+P       Q  sig_pq bytes
     ──────────── [header ends at offset header_len]
80+N+M+P+Q     R  encrypted_payload: nonce(12) || tag(16) || ciphertext(R-28 bytes)
```

The variable names N, M, P, Q, R denote byte lengths and are encoded as the `*_len`
fields listed above.

### 4.2 TrayID Byte

```
0x01 = level2-25519   (X25519 + Kyber512 + Ed25519 + Dilithium2)
0x02 = level2         (P-256  + Kyber512 + ECDSA P-256 + Dilithium2)
0x03 = level3         (P-384  + Kyber768 + ECDSA P-384 + Dilithium3)
0x04 = level5         (P-521  + Kyber1024 + ECDSA P-521 + Dilithium5)
```

### 4.3 KEM Step

Identical to OBIWAN §3.2 and §3.3: encapsulate with the KEM-classical slot and the
KEM-PQ slot to obtain (CT_cl, SS_cl) and (CT_pq, SS_pq).

### 4.4 HYKE Key Derivation

HYKE uses its own KMAC256 invocation with a distinct custom string and **no** `len32`
length prefixes in the message:

```
key_param  = SS_cl
msg        = SS_pq || CT_cl || CT_pq || salt   (raw concatenation)
custom_str = "obi-wan-hybrid-sig-v1"             (21 bytes)

sym_key = KMAC256(key=key_param, msg=msg, output_length=256 bits, custom=custom_str)
```

`salt` is 32 random bytes chosen fresh at sign time and stored in the header.

### 4.5 Symmetric Encryption

HYKE always uses **AES-256-GCM** (key = sym_key from §4.4).
No additional data (AAD) is used.
Output layout: nonce(12) || tag(16) || ciphertext (identical to §3.5).

### 4.6 Context Binding

To prevent key-substitution attacks, a 64-byte context value is derived that commits to
both KEM public keys:

```
msg_ctx    = pk_pq || "obi-wan-hybrid-sig-v1"   (PQ pk bytes, then domain string raw)
custom_ctx = ""                                   (empty custom string)

ctx = KMAC256(key=pk_cl, msg=msg_ctx, output_length=512 bits, custom=custom_ctx)
```

`ctx` is 64 bytes.

### 4.7 Partial Header (Signed Region)

The **partial header** is the prefix of the full wire header up to (and including) the
CT bytes, but **not** the signature bytes.  It commits to all header metadata and both
KEM ciphertexts.  The signature length fields *are* included so signatures cannot be
swapped for shorter or longer ones from another session.

```
partial_header = Magic(4)
              || Version(2)
              || TrayID(1)
              || Flags(1)
              || header_len(4)     -- includes P + Q
              || payload_len(4)
              || tray_uuid(16)
              || salt(32)
              || ct_classical_len(4)
              || ct_pq_len(4)
              || sig_classical_len(4)
              || sig_pq_len(4)
              || CT_classical(N)
              || CT_pq(M)
```

Total length: 80 + N + M bytes.

### 4.8 Message to Sign

```
m_to_sign = ctx(64) || partial_header(80+N+M) || encrypted_payload(R)
```

Both classical and PQ signatures are computed over the identical `m_to_sign`.

### 4.9 Signature Algorithms

**Ed25519** (level2-25519)
- OpenSSL raw key API, `NULL` message digest (Ed25519 hashes internally per RFC 8032).
- One-shot sign/verify (`EVP_DigestSign` / `EVP_DigestVerify`).
- Signature: 64 bytes (fixed).

**ECDSA** (level2, level3, level5)
- Digest: SHA-256 (P-256), SHA-384 (P-384), SHA-512 (P-521).
- Wire format: **IEEE P1363** — raw big-endian r || s, zero-padded to fixed size.
  - P-256: 64 bytes (32 + 32).
  - P-384: 96 bytes (48 + 48).
  - P-521: 132 bytes (66 + 66).
- OpenSSL produces DER; implementation converts DER → P1363 for storage and P1363 → DER
  for verification.

**Dilithium** (all hybrid levels)
- Reference implementation: `pqcrystals_dilithium{2|3|5}_ref_signature` /
  `pqcrystals_dilithium{2|3|5}_ref_verify`.
- No context string (ctx = NULL, ctx_len = 0).
- Signature sizes: Dilithium2 = 2420 B, Dilithium3 = 3309 B, Dilithium5 = 4627 B.
  > These are the sizes from `dilithium/ref/api.h` (ref implementation); they may
  > differ from the ML-DSA FIPS 204 specification.

### 4.10 Sign Procedure

1. Load tray; locate all four slots.  Signing requires Sig-classical sk and Sig-PQ sk.
2. Read plaintext from input file.
3. Classical KEM encapsulation → (CT_cl, SS_cl).
4. Kyber encapsulation → (CT_pq, SS_pq).
5. Generate 32 random bytes → salt.
6. Derive sym_key = HYKE-KMAC256(SS_cl, SS_pq, CT_cl, CT_pq, salt) per §4.4.
7. AES-256-GCM encrypt plaintext → encrypted_payload = nonce || tag || ct.
8. Compute expected sig sizes P and Q from the tray type (known, fixed per level).
9. Build partial_header per §4.7 (header_len includes P + Q).
10. Compute ctx = context-binding per §4.6.
11. Build m_to_sign = ctx || partial_header || encrypted_payload.
12. sig_classical = Sign_classical(Sig-classical sk, m_to_sign).
13. sig_pq = Dilithium.Sign(Sig-PQ sk, m_to_sign).
14. Serialise full wire bytes:
    - partial_header || sig_classical(P) || sig_pq(Q) || encrypted_payload(R).
15. Base64-encode (64-character line wrap); wrap in armor (see §4.11).

### 4.11 Verify Procedure

1. Load tray; locate all four slots.  Verify requires KEM-classical sk and KEM-PQ sk.
2. Strip armor from input; base64-decode to wire bytes.
3. Parse wire header per §4.1 to recover all fields.
4. Check TrayID byte matches the loaded tray type; abort if not.
5. Reconstruct partial_header (same rule as §4.7, using parsed field values).
6. Compute ctx = context-binding per §4.6 using the KEM public keys from the tray.
7. Build m_to_sign = ctx || partial_header || encrypted_payload.
8. Verify classical signature: Verify_classical(Sig-classical pk, m_to_sign, sig_classical).
   Abort with error if invalid.
9. Verify PQ signature: Dilithium.Verify(Sig-PQ pk, m_to_sign, sig_pq).
   Abort with error if invalid.
10. Classical KEM decapsulation: SS_cl = Decaps(KEM-classical sk, CT_cl).
11. Kyber decapsulation: SS_pq = Kyber.Dec(KEM-PQ sk, CT_pq).
12. Derive sym_key = HYKE-KMAC256(SS_cl, SS_pq, CT_cl, CT_pq, salt from header).
13. AES-256-GCM decrypt encrypted_payload → plaintext.
14. Write plaintext to output.

### 4.12 Armor Format

```
-----BEGIN HYKE SIGNED FILE-----
<base64 lines, 64 characters each>
-----END HYKE SIGNED FILE-----
```

## 5. Key and Ciphertext / Signature Sizes

### 5.1 KEM Algorithms

| Algorithm  | Public key | Secret key | Ciphertext | Shared secret |
|------------|-----------|-----------|-----------|--------------|
| X25519     | 32 B      | 32 B      | 32 B      | 32 B         |
| P-256      | 65 B      | 32 B      | 65 B      | 32 B         |
| P-384      | 97 B      | 48 B      | 97 B      | 48 B         |
| P-521      | 133 B     | 66 B      | 133 B     | 66 B         |
| Kyber512   | 800 B     | 1632 B    | 768 B     | 32 B         |
| Kyber768   | 1184 B    | 2400 B    | 1088 B    | 32 B         |
| Kyber1024  | 1568 B    | 3168 B    | 1568 B    | 32 B         |

EC public keys are uncompressed points (0x04 || X || Y).
EC secret keys are big-endian scalars.
EC ciphertexts are the ephemeral public key (same format as the public key).

### 5.2 Signature Algorithms

| Algorithm    | Public key | Secret key | Signature   | Format   |
|--------------|-----------|-----------|------------|---------|
| Ed25519      | 32 B      | 32 B      | 64 B       | fixed    |
| ECDSA P-256  | 65 B      | 32 B      | 64 B       | P1363 r\|\|s |
| ECDSA P-384  | 97 B      | 48 B      | 96 B       | P1363 r\|\|s |
| ECDSA P-521  | 133 B     | 66 B      | 132 B      | P1363 r\|\|s |
| Dilithium2   | 1312 B    | 2560 B    | 2420 B     | variable ≤ max |
| Dilithium3   | 1952 B    | 4032 B    | 3309 B     | variable ≤ max |
| Dilithium5   | 2592 B    | 4896 B    | 4627 B     | variable ≤ max |

Dilithium signature sizes are from the reference implementation (`dilithium/ref/api.h`).
The HYKE format pre-declares the maximum signature sizes in the header length fields and
stores the actual (possibly shorter) signature bytes; the `sig_{classical,pq}_len` fields
in the wire header reflect the actual sizes written.

## 6. Implementation Notes

- **Byte order**: all multi-byte integers in both wire formats are big-endian.
- **XKCP lengths in bits**: the XKCP `KMAC256()` and `SHAKE256()` functions expect key,
  message, output, and custom lengths in **bits**.  Multiply byte counts by 8 before
  passing them.
- **OBIWAN vs HYKE KDF**: OBIWAN-SHAKE and OBIWAN-KMAC use `len32()` length prefixes on
  every input.  The HYKE KDF uses raw concatenation with no length prefixes.  The HYKE
  context binding also uses raw concatenation.
- **Encrypt-then-sign ordering**: in HYKE, the payload is fully encrypted before either
  signature is computed.  Signatures cover the ciphertext, not the plaintext.
- **ECDSA P1363**: signatures are stored as fixed-length raw r || s.  Sign converts
  OpenSSL's DER output to P1363; Verify converts P1363 back to DER before calling OpenSSL.
- **Dilithium context string**: the Dilithium reference API supports a context string
  parameter; obi-wan passes NULL / length 0 (no context string used).
- **Tray UUID in HYKE**: the 36-character UUID string (xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx)
  is stored as 16 raw bytes (dashes stripped, hex pairs decoded).
- **Tray UUID derivation**: the tray UUID is derived from the public keys via BLAKE3
  key-derivation mode (see `pq/UUID-DERIVATION.md`); it is verified on tray load.
