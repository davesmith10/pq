 
 # Defending Certain Choices - XKCP Pros and Cons
 
 ## XKCP in libcrystals-1.2
 
   Yes, XKCP is a dependency. It is a dynamic (shared) library — libXKCP.so. This adds an additional shared library to deploy.
 
 ### What it's used for
 
   XKCP provides the Keccak/SHA-3 family primitives. In libcrystals-1.2, the sole consumer is src/kdf.hpp, which calls
   two functions from it:
 
 
   +------------------------------------------------------------------------------------------------------+
   ¦        XKCP function        ¦                                Used for                                ¦
   +-----------------------------+------------------------------------------------------------------------¦
   ¦ SHAKE256() (from            ¦ OBIWAN SHAKE-256 KDF — hashes the two shared secrets + ciphertexts     ¦
   ¦ SimpleFIPS202.h)            ¦ into a 32-byte encryption key                                          ¦
   +-----------------------------+------------------------------------------------------------------------¦
   ¦ KMAC256() (from             ¦ OBIWAN KMAC KDF, HYKE KDF, and HYKE context binding (compute_hyke_ctx) ¦
   ¦ SP800-185.h)                ¦                                                                        ¦
   +------------------------------------------------------------------------------------------------------+
 
   These are the only XKCP calls — everything else (Kyber's internal SHAKE, Dilithium's internal Keccak) uses the
   statically-compiled fips202_ref archives, which are entirely separate and do not go through XKCP.
 
   Where the library lives
 
   +----------------------------------------------------------------------------------------------------+
   ¦    Artifact    ¦                                       Path                                        ¦
   +----------------+-----------------------------------------------------------------------------------¦
   ¦ Pre-built .so  ¦ XKCP/bin/x86-64/libXKCP.so (source of truth)                                      ¦
   +----------------+-----------------------------------------------------------------------------------¦
   ¦ Installed copy ¦ /usr/local/lib/libXKCP.so (put there by install.sh)                               ¦
   +----------------+-----------------------------------------------------------------------------------¦
   ¦ Headers        ¦ XKCP/bin/x86-64/libXKCP.so.headers/ (includes SimpleFIPS202.h, SP800-185.h, etc.) ¦
   +----------------------------------------------------------------------------------------------------+
 
 ### Why it stays dynamic (not bundled into the fat archive)
 
   The install.sh explicitly copies libXKCP.so to ${PREFIX}/lib but does not extract it into the fat archive. This is
   intentional — XKCP is a pre-built .so with no accompanying .a, so it can't be merged. The CrystalsConfig.cmake
   generated by install.sh does a find_library(_xkcp XKCP HINTS ${prefix}/lib REQUIRED) and adds it as a transitive link
   dep of Crystals::crystals.
 
   The RPATH on both scotty and obi-wan includes /usr/local/lib specifically so the runtime linker finds libXKCP.so there
   without needing LD_LIBRARY_PATH.

## Can XKCP be replaced?

  - SHAKE256 — yes, two viable internal sources.
  - KMAC256 — yes, via OpenSSL, but with caveats.

###  SHAKE256

  Two implementations are already statically linked in the fat archive:

  +-----------------------------------------------------------------------------------------------------------------+
  ¦              Source              ¦                Symbol name                ¦           One-shot API           ¦
  +----------------------------------+-------------------------------------------+----------------------------------¦
  ¦ pqcrystals_kyber_fips202_ref     ¦ pqcrystals_kyber_fips202_ref_shake256     ¦ shake256(out, outlen, in, inlen) ¦
  +----------------------------------+-------------------------------------------+----------------------------------¦
  ¦ pqcrystals_dilithium_fips202_ref ¦ pqcrystals_dilithium_fips202_ref_shake256 ¦ same                             ¦
  +-----------------------------------------------------------------------------------------------------------------+

  The XKCP call in kdf.hpp is:
  SHAKE256(key.data(), 32, buf.data(), buf.size())

  The Kyber/Dilithium equivalent (after accounting for the FIPS202_NAMESPACE macro mangling) is:
  pqcrystals_kyber_fips202_ref_shake256(key.data(), 32, buf.data(), buf.size())

  The signatures are functionally identical — direct drop-in. No additional dependency, already in the archive.

 ### KMAC256

  No KMAC implementation exists in the Kyber or Dilithium ref code — they only implement FIPS 202 (SHAKE/SHA3), not SP 800-185 (cSHAKE/KMAC).

  However, OpenSSL 3.x is already a link dep and it has KMAC256 via EVP_MAC:

  EVP_MAC_fetch(NULL, "KMAC256", NULL)   // returns EVP_MAC*

  The downside: the OpenSSL EVP_MAC API is considerably more verbose than the XKCP one-liner. The XKCP call is a single function:
  KMAC256(key, keyBitLen, msg, msgBitLen, output, outputBitLen, custom, customBitLen)

  The OpenSSL equivalent requires EVP_MAC_CTX_new, setting OSSL_MAC_PARAM_KEY, OSSL_MAC_PARAM_CUSTOM, EVP_MAC_init, EVP_MAC_update,
  EVP_MAC_final, EVP_MAC_CTX_free — roughly 15–20 lines of boilerplate per call. There are three KMAC256 call sites in kdf.hpp.

  Also note: OpenSSL's KMAC256 takes byte lengths, not bit lengths (unlike XKCP which takes bit lengths for key, message, output, and custom
  string) — so the call conventions differ and the conversion would need care.

  ¦          ¦ archive)                                                ¦                                                                   ¦
  +----------+---------------------------------------------------------+-------------------------------------------------------------------¦

### Why We're Not Doing It

In part because the XKCP is a independent suppyler of the algorithms, and in part because we may have additional uses for the library later.
